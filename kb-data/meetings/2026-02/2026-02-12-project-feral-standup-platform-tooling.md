## Executive Summary

**TL;DR:** The team formally adopted the hybrid component library approach (CLT for cross-brand, ShadCN for brand-specific), agreed to set up Storybook in the Beans frontend repo for isolated iteration, and identified critical gaps in web analytics tooling (surveys, heat maps, AB testing) that need resolution across BlueConic and Contentful.

**Overview:** The standup resolved the component library decision from the previous session — the team will use a hybrid approach, leveraging the company component library (CLT) for cross-brand primitives while building Beanz-specific composite components locally using ShadCN with Tailwind. Josh is setting up a Storybook instance in the Beans frontend repo to enable visual iteration. Easwar highlighted the need for CMS content model documentation so agents can understand Contentful attributes. The team discussed significant gaps in web analytics: surveys lack a standardized BRG-wide tool, heat maps and replays need clarification on existing capabilities, and AB testing sits between BlueConic (CDP) and Contentful with unclear delineation. Prasanna reported on setting up the Databricks replication of Justin's Datasette analytics platform, with data access blockers for Algolia and Salesforce. Justin presented the full data architecture — from data pipelines through the semantic layer to the [[intelligence-platform|Intelligence Platform]] — and announced plans to bring the Marketing Cloud team into conversations for real-time email personalization. Josh confirmed the KB repo migration to Revel Digital with security guardrails and PR review processes.

**Key Takeaways:**
- Hybrid component library approach formally adopted — CLT for cross-brand, ShadCN locally for Beanz-specific needs
- Storybook instance in Beans frontend repo unlocks visual component iteration; team to start onboarding immediately
- Web analytics has significant gaps: no standardized survey tool, unclear heat map/replay capabilities, AB testing split across BlueConic and Contentful
- Datasette analytics platform is Justin's proof of concept; Prasanna replicating into Databricks for production with cross-validation planned
- KB migration to Revel Digital happening with added security guardrails and category-based PR review

## Decisions

### Decision 1: Hybrid Component Library Approach

| Field | Detail |
|-------|--------|
| **Decision** | Adopt hybrid approach: use CLT (company library) for cross-brand/cross-repo primitives, build brand-specific composite components locally in the Beans repo using ShadCN with Tailwind |
| **Rationale** | Cross-brand primitives (buttons, text, cards) ensure visual consistency and multi-tenancy support, while brand-specific components (quiz, PDP, buy stack) need faster iteration locally. ShadCN is just raw JSX with Tailwind — the only addition is Tailwind support in the frontend repo. |
| **Alternatives Considered** | Full CLT integration only (too slow for Beanz iteration), full breakaway with ShadCN only (loses cross-brand consistency) |
| **Stakeholders** | Josh, Easwar, Santhosh, Sophie, Justin, Rishab (consulted), Tal (to be informed) |
| **Confidence** | 1.0 |

### Decision 2: Storybook for Visual Component Iteration

| Field | Detail |
|-------|--------|
| **Decision** | Set up a Storybook instance within the Beans frontend repo for isolated visual component iteration, with mock data contracts for parallel front-end/back-end development |
| **Rationale** | Provides a sandbox for iterating on composite components (quiz, collection UI) without needing the full CMS pipeline. Allows multiple team members to prototype variant stories against the same mock data. |
| **Alternatives Considered** | Iterating directly in the full application (requires CMS integration), Figma-only prototyping (no code output) |
| **Stakeholders** | Josh, Easwar, Sophie |
| **Confidence** | 1.0 |

## Argument Maps

### Debate 1: Self-Managing KB vs Human Review Gates

**Claim:** *The knowledge base should be allowed to self-manage via agentic workflows at scale*

**Supports:**

| Argument | Speaker |
|----------|---------|
| It's not possible to manually manage at the scale content is being created | Justin |
| AI-driven meeting summaries and task tracking are already working in test runs | Justin |

**Attacks:**

| Argument | Speaker |
|----------|---------|
| Architecture and system design content needs human eyes — if it becomes source of truth, we must understand what's in it | Josh |
| Need process gates and baby steps first, especially for architectural content | Josh |
| Need to validate agent-generated data before giving stakeholders access | Raam |

**Resolution:** Compromise — adopt agentic workflows with human review gates. Category-based PR review ensures architectural content gets technical review. Non-technical areas can be more flexible. Revisit in ~6 months.

### Debate 2: Datasette (Proof of Concept) vs Databricks (Production)

**Claim:** *Justin's Datasette analytics platform should be transitioned to Databricks for production use*

**Supports:**

| Argument | Speaker |
|----------|---------|
| Production system needs Databricks — 21M rows is OK but 200M won't scale in SQLite | Raam |
| REST APIs consumed by the portal should come from Databricks, not Datasette | Raam |
| Existing tools and infrastructure should be leveraged where possible | Raam |

**Attacks:**

| Argument | Speaker |
|----------|---------|
| Datasette allowed 12 months of rapid iteration that would have been blocked waiting for analytics team infrastructure | Justin |
| Built-in MCP, data dictionary, semantic layer, and LLM enrichment pipelines already working | Justin |
| Running both in parallel enables cross-validation of business data | Justin |

**Resolution:** Both run in parallel for now. Prasanna replicates into Databricks. Cross-validation will surface data gaps in both directions. Datasette remains Justin's exploration playground; Databricks becomes the production path.
