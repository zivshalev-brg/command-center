## Argument Maps

### Debate 1: Figma vs Code-First Prototyping

**Claim:** *The team should prototype directly in code (Claude Code) rather than starting with Figma designs*

**Supports:**

| Argument | Speaker |
|----------|---------|
| With a good system design framework and Claude Code, you can go from idea to front-end without an intermediate Figma step | Ziv Shalev |
| The starting point is an idea — anyone should be able to progress it to production; we need T-shaped people, not UX dependency | Justin Le Good |
| UXPilot and similar tools can't reliably generate Figmas that adhere to a specific design system, so Figma may still require manual design work | Ziv Shalev |
| Working in code with Claude Code means the prototype is already in the codebase, closer to production | Justin Le Good |

**Attacks:**

| Argument | Speaker |
|----------|---------|
| Figma is faster for pixel-perfect designs — Justin's prototype likely took multiple days, whereas Figma mockups are quicker | Raam Baranidharan |
| Figma enforces brand guidelines, visual guidelines, and design consistency more easily than code prototyping | Raam Baranidharan |
| Figma doesn't require the UX team — anyone can use it, and it's a proven tool for acceleration | Raam Baranidharan |
| Using Figma doesn't mean reverting to old processes — it's about using purpose-built tools for speed | Raam Baranidharan |

**Resolution:** Ziv proposed testing both approaches in parallel for two weeks, then comparing results. The team agreed.

### Debate 2: Component Library — Use, Hybrid, or Break Away

**Claim:** *Beanz should break away from the platform component library to enable faster front-end iteration*

**Supports:**

| Argument | Speaker |
|----------|---------|
| Most current Beanz components are already custom — the site barely uses the component library today | Easwar |
| Platform component library components are stationary (structure is static) — moving a badge from one side to another won't work | Josh Mu |
| Beanz-unique features (buy stack, questionnaire, exit survey, collections) aren't in the component library and are exactly what needs iteration | Ziv Shalev |
| Full flexibility requires breaking away from component library constraints | Josh Mu |

**Attacks:**

| Argument | Speaker |
|----------|---------|
| Breaking away means the user experience won't be seamless across the top-of-funnel and my-account/cart/checkout (which stays on the platform) | Josh Mu |
| Even with a breakaway, a Beanz-branded component library will still be needed for consistency within the top-of-funnel stack | Josh Mu |

**Resolution:** Pending. Rishab to present a decision framework (hybrid, breakaway, or full integration) by tomorrow or early next week.
