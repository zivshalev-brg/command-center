## Argument Maps

### Debate 1: Dev Environment Value Proposition

**Claim:** *Is the dev environment worth maintaining when the backend lacks a dev tier?*

**Supports (Keep Dev):**

| Argument | Speaker |
|----------|---------|
| Dev needed for new Contentful features while QA/stage maintains existing website | Easwar |
| From SDLC standpoint, you still need a dev instance — can't develop on UAT | Santhosh |
| Dev will be used for the new feature stack during migration period | Easwar |

**Supports (Remove Dev):**

| Argument | Speaker |
|----------|---------|
| If dev points to QA backend, it's an extra step that does nothing | Josh |
| Other teams leverage QA directly and don't really use dev | Josh |
| Backend doesn't have dev for all services — front end is doing mix-and-match of environments | Josh |
| Multiple front-end environments pointing to same backend (many-to-one) adds confusion | Santhosh |

**Resolution:** Keep dev for now. Easwar will copy QA configs to dev for immediate unblocking. The broader value proposition question remains open — team acknowledges the need during the Contentful migration transition period.

### Debate 2: Survey Tool — Hotjar vs Custom Exit Survey

**Claim:** *Should Hotjar be adopted for all survey needs including exit surveys, or should the existing custom AEM/Salesforce component be preserved?*

**Supports (Hotjar):**

| Argument | Speaker |
|----------|---------|
| Hotjar already approved and previously used for gift card feedback and other features | Santhosh |
| Provides more than surveys — heat maps, recordings, playbacks, journeys | Santhosh |
| Standardized tool across BRG; build option only when no tool exists in ecosystem | Santhosh |
| Need event-triggered surveys at various touchpoints (post-order, post-feature usage) that current custom component doesn't support | Justin |

**Supports (Keep Custom / Cautious Approach):**

| Argument                                                                                | Speaker  |
| --------------------------------------------------------------------------------------- | -------- |
| Current exit survey already integrates with Salesforce — established pattern            | Justin   |
| Adding another platform may increase cost significantly for functionality we don't need | Ziv      |
| May be over-complicating something that works; need to understand pros/cons first       | Ziv      |
| Previous Hotjar usage was turned off due to licensing costs                             | Santhosh |

**Resolution:** Pending. Hotjar confirmed for general survey/analytics use (heat maps, recordings). Exit survey specifically requires further review — Ziv requested a requirements document to evaluate whether Hotjar is the right solution or the existing Salesforce integration should be preserved and migrated.

### Debate 3: KB/Dashboard Hosting — Obsidian vs Jira/Cloud

**Claim:** *Should the AI project management dashboard and KB remain as a local Obsidian vault, or be hosted/integrated with Jira?*

**Supports (Keep Obsidian/GitHub):**

| Argument | Speaker |
|----------|---------|
| Agent-first architecture with strict content quality gates — all content processed by Claude skills | Justin |
| PR-based review process already operational with assigned reviewers per category | Josh |
| Testing the envelope of AI-first project management, not just code writing | Ziv |
| Building centralized context that can be leveraged downstream | Ziv |

**Supports (Move to Jira/Cloud):**

| Argument | Speaker |
|----------|---------|
| Everybody already has Jira access — no GitHub dependency | Raam |
| Should be persistent with logins if there's benefit to hosting it | Ziv |
| Assumes GitHub access, which may not scale to other departments | Raam |

**Resolution:** Keep Obsidian/GitHub for now. Jira could act as a "skin" — JIRA would display information but everything still operates from the KB repo. Justin to provide team onboarding walkthrough. If the approach works well, team will evaluate hosting options for broader access.
