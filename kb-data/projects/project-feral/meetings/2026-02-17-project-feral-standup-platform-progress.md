## Decisions

### Decision 1: Vendor Data Stored in Americano DynamoDB (Not PIM)

| Field | Detail |
|-------|--------|
| **Decision** | Roaster/vendor address data will be stored by extending the existing Beanz Connect (Americano team) DynamoDB table. Vendor data will NOT go into PIM or be authored in Contentful. |
| **Rationale** | PIM and CMS are hit too frequently by downstream systems (Salesforce, BLP, Algolia, CT). The Americano DynamoDB service already holds vendor address data from its existing use. Extending it avoids creating a new source. Middleware pipelines will pull from this service (addresses, lat/long, SKU associations) rather than from PIM. Long-term: D365 is the source of vendor IDs and may get a UI for reporting. |
| **Alternatives Considered** | PIM (too many systems hitting it constantly); Contentful CMS (not editable for operational address data); new dedicated store (unnecessary given existing service) |
| **Stakeholders** | Santhosh, Nagesh, Santosh Sabat, Sophie Thevenin |
| **Confidence** | 0.9 (high-level decided; detailed flow refinement call still needed with Sahin present) |

### Decision 2: Roaster Address Authoring in CMS — Ruled Out

| Field | Detail |
|-------|--------|
| **Decision** | Roaster address data will NOT be authored in Contentful CMS. Addresses must come from source (DynamoDB/Beanz Connect). Roaster descriptions and other web attributes will go into Contentful, but deferred 6 months (not in current scope). |
| **Rationale** | Address data is operational, not editorial — it belongs at source. Algolia will receive location data (lat/long) from the DynamoDB service for roaster listing page filtering. |
| **Alternatives Considered** | Authoring addresses in CMS (ruled out definitively); mixing address and editorial data in Contentful (deferred) |
